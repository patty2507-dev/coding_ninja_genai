Chapter 02

Reactive vs Deliberative

The two oldest, simplest architectures in AI — and CodeSentinel already has a piece of each, whether we named it or not.

⚡
Reactive

No world model, no memory, no planning. A direct rule: condition seen → fixed action taken. Instant, cheap, can't be talked out of it.

🧠
Deliberative

Builds a model of the situation, sets a goal, plans a sequence of steps, then acts. Slower, costlier, but can handle novel situations the reactive rule never anticipated.

The Reactive layer CodeSentinel doesn't have yet — but should

The gap

Right now, if a diff contains a hardcoded AWS key, CodeSentinel finds out about it the SAME way it finds out about a missing docstring — by the LLM reading get_file_content output and reasoning about it. That's a Deliberative response to something that should be Reactive: instant, zero-reasoning, can't-be-argued-with.

reactive_guard.py NEW — Session 4 
# Runs BEFORE the agent loop even starts — zero LLM calls, zero reasoning.
# This is a pure Reactive architecture: condition -> fixed action.
import re

SECRET_PATTERNS = [
    r"AKIA[0-9A-Z]{16}",               # AWS access key
    r"sk-[a-zA-Z0-9]{32,}",             # OpenAI-style secret key
    r"-----BEGIN PRIVATE KEY-----",
]

def reactive_guard(diff_text: str) -> dict:
    """No LLM call. No memory read. No plan. Just: pattern found -> block."""
    for pattern in SECRET_PATTERNS:
        if re.search(pattern, diff_text):
            return {"blocked": True, "reason": f"matched {pattern}"}
    return {"blocked": False}

The Deliberative part CodeSentinel already had

Every time the agent calls get_pr_details, then get_pr_files, then reasons about which files matter before calling get_file_content — that's Deliberative behaviour. It built an internal picture (world model) of the PR before acting. We just never named it.

Reactive alone — fails

Would catch the hardcoded key. Would completely miss "this PR introduces an N+1 query" — that needs reasoning over multiple files, which no fixed rule can express.

Deliberative alone — fails differently

Would eventually catch the hardcoded key too — but only after several LLM calls and real latency, for something that needed zero intelligence to catch.

The actual fix

Neither replaces the other. Put the Reactive guard in front, as a fast, cheap, non-negotiable filter. Let the Deliberative agent loop handle everything the guard doesn't block. This pairing is called a Layered architecture — Chapter 04.

Chapter 03

BDI — Beliefs, Desires, Intentions

This is the piece that maps directly onto Session 3's memory tables — almost embarrassingly directly.

💭
Beliefs
What the agent currently thinks is true — pulled straight from search_memory results
🎯
Desires
Goals it holds — possibly in tension with each other
✅
Intentions
The committed plan of action for this specific PR
Mapped to the real PR #11 run

Belief: search_memory(kind="author_history", author="rahul8879") returns "this author left a SQL injection risk in PR #10, same file pattern." That's a belief, literally fetched from Postgres.

Desire (conflict): Desire A = "flag every recurring risk, no matter how minor." Desire B = "don't slow down a hotfix PR tagged urgent." These two desires actively disagree on PR #11 if it's also tagged urgent.

Intention: the resolved, committed choice — e.g. "post a comment referencing the pattern, but don't block the merge." This is what post_pr_comment actually sends, after the conflict above got resolved one way.

The uncomfortable part

Right now, CodeSentinel has no explicit Desire-conflict step. It just... goes with whatever the single LLM call decides in the moment, with no visibility into which desire won. That's not wrong, but it's not principled either — there's no resolve_conflict() anywhere in the trace.

bdi_intention.py NEW — Session 4 
# Makes the Desire conflict EXPLICIT instead of buried inside one LLM call.
def form_intention(beliefs: dict, pr_labels: list[str]) -> str:
    urgent = "urgent" in pr_labels
    has_recurring_risk = beliefs.get("author_history") is not None

    # Desire A: flag every recurring risk   Desire B: don't slow down urgent PRs
    if has_recurring_risk and urgent:
        return "comment_dont_block"   # explicit resolution, not implicit
    elif has_recurring_risk:
        return "comment_and_request_changes"
    else:
        return "standard_review"

This single function is what separates "BDI as a diagram on a slide" from "BDI actually implemented" — it's the one place in the whole codebase where you can point and say this is where competing goals get weighed against each other.

Fix #1 — Tiered approval instead of one gate

The actual change to the graph

Instead of one binary "approve post_pr_comment or don't," every tool now carries a risk tier. Only Tier 2 tools pause for human approval — same as before for post_pr_comment, but now save_finding on a NEW author (never seen before, no prior beliefs to check against) also gets gated, since a wrong first entry poisons every future review of that person.

risk_tiers.py NEW — Session 4 
# Replaces the single hardcoded "only post_pr_comment needs approval" rule
# from Session 3 with a tiered lookup, checked in the graph's interrupt condition.
RISK_TIER = {
    "get_pr_details":      0,  # read-only, public data, no oversight needed
    "get_pr_files":        0,
    "get_file_content":    0,
    "search_memory":       1,  # NEW: elevated when kind == "author_history"
    "save_finding":        1,  # elevated only for first-time authors, see below
    "post_pr_comment":     2,  # unchanged from Session 3 — always gated
}

def needs_approval(tool_name: str, tool_args: dict, seen_authors: set) -> bool:
    tier = RISK_TIER.get(tool_name, 2)  # unknown tool defaults to MOST cautious tier
    if tool_name == "search_memory" and tool_args.get("kind") == "author_history":
        return True   # Fix #2 — see below
    if tool_name == "save_finding" and tool_args.get("author") not in seen_authors:
        return True  # first-ever finding on this person — worth a human glance
    return tier >= 2

Fix #2 — The profiling concern, named and handled

Say this part out loud in class

search_memory(kind="author_history", author="rahul8879") pulling up someone's mistake history, silently, every single PR, with no log a human ever sees — that's the exact shape of "quiet profiling," even with entirely good intentions behind it. The fix above doesn't remove the capability. It makes every use of it visible and consent-gated, which is the honest trade-off: slightly slower, meaningfully more accountable.

Is this actually different from just "more if-statements"?

What it looks like

Yes — mechanically, needs_approval() is an if-statement. Layered architecture isn't a new algorithm; it's a naming convention for WHERE control decisions live, so a team of engineers can reason about them explicitly instead of them being buried inside one giant prompt.

Why it's worth naming anyway

Before this session, "should this be gated?" was answered implicitly by the LLM's judgment on that one turn. Now it's answered by a testable, versioned Python function that a code reviewer can actually review — that's the entire value of calling it an architecture instead of a vibe.

CodeSentinel has three of CoALA's four memory modules for real, in production code. Chapter 06

Design Patterns — Planning, ReAct, Tool Use, Reflection

Architectures are the blueprint. These four are the reusable techniques living INSIDE that blueprint — and CodeSentinel already uses two of them, whether we named it or not.

Planning — the pattern CodeSentinel does NOT currently use, and why that's fine

Planning means: decide the full sequence of steps upfront, execute in fixed order, no re-thinking mid-way. CodeSentinel never does this — every PR gets the same free-form agent → tools → agent loop, whether it's a one-line README fix or a five-file security-sensitive change.

The cost of not having this

A trivial docs-only PR still pays for a full reasoning loop — the LLM decides, on its own, that it only needs get_pr_files and can skip get_file_content. That's usually the right call, but it's non-deterministic: two identical docs-only PRs could take a different number of LLM calls.

planning_fast_path.py NEW — Session 4, optional optimization 
# Fixed plan for the ONE case we know in advance: docs-only PRs.
# Skips the agent loop entirely -- cheaper, deterministic, predictable.
def is_docs_only(pr_files: list[str]) -> bool:
    return all(f.endswith((".md", ".txt", ".rst")) for f in pr_files)

def docs_only_plan(pr_number: int) -> list[str]:
    # fixed, 2-step plan -- never calls the full agent loop
    return ["get_pr_files", "post_pr_comment"]

ReAct — the pattern CodeSentinel has been running since Session 2, unnamed

ReAct = Thought → Action → Observation, looped, until the model decides it has enough to stop. That IS the agent → tools → agent loop, verbatim. We're not adding this pattern — we're finally naming what's already there.

react_loop.py Same loop as Session 2/3 — annotated as ReAct 
from langgraph.graph import StateGraph, END

def agent_node(state):
    # THOUGHT: the LLM reasons over messages so far, decides next ACTION
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def tools_node(state):
    # ACTION -> OBSERVATION: run whichever tool the LLM picked, feed result back
    last = state["messages"][-1]
    results = [run_tool(call) for call in last.tool_calls]
    return {"messages": results}

graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tools_node)
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue, {"tools": "tools", "end": END})
graph.add_edge("tools", "agent")  # <- THIS edge is the entire ReAct pattern
The one-line answer to give if asked in an interview

"CodeSentinel's core loop is a ReAct agent — its architecture is CoALA-shaped (real episodic/semantic/procedural memory), with a Reactive guardrail layer in front and a tiered BDI-style approval system deciding what needs a human."

Tool Use — what Session 2/3's tools got right (and the one thing to fix)

get_pr_details, get_pr_files, get_file_content, post_pr_comment, search_memory, save_finding — six tools, each single-purpose, each with a clear docstring the LLM reads to decide relevance. That's good tool design, already.

The one gap

save_finding has no code-level duplicate check — flagged already in Session 3 as an open risk. Good tool design also means the tool itself should refuse bad calls, not rely purely on the docstring saying "only if genuinely new."

Reflection — the pattern we're adding tonight, for real

Right now, whatever post_pr_comment drafts goes straight to human approval with zero self-critique. Reflection adds one more LLM call BEFORE that approval step — a separate "critic" pass, checking tone, missed issues, and false confidence.

reflect_before_posting.py NEW — Session 4 
# Slots in right before the existing post_pr_comment approval gate.
def critique_draft(draft_comment: str, diff: str) -> dict:
    prompt = f"""You are a senior engineer critiquing a junior reviewer's PR comment.
Diff: {diff}
Draft: {draft_comment}
Check: tone too harsh/soft? missed security issue? vague, no line refs?
If good enough to post, reply exactly APPROVED. Otherwise list fixes."""
    result = llm.invoke(prompt)
    approved = result.content.strip().upper().startswith("APPROVED")
    return {"approved": approved, "critique": result.content}

# Graph change: draft -> critique -> (approved? go to human gate : refine -> critique)
# Capped at 2 rounds -- matches the "over-analysis paralysis" weakness we
# already flagged as an open risk for save_finding duplicates in Ch.05.
Where the critique gets stored

If a round is rejected, the critique itself gets written to save_finding's procedural namespace — so the SAME class of drafting mistake (e.g. "too harsh in tone") doesn't need to be re-caught by the critic every single time. That's Reflection feeding directly back into Session 3's memory system.

Why this shape, not something fancier

The Reactive guard stays first because it's the cheapest possible check and must never be skippable by clever prompting. The fast-path router avoids paying ReAct's latency tax on trivial PRs. Everything else — BDI, Reflection, tiered approval — sits AFTER the agent loop rather than inside it, because each of those is a genuinely separate concern (goal conflict, self-critique, risk gating) and mixing them into one giant prompt is exactly the "one opaque LLM call" problem Chapter 05 called out.

CodeSentinel Has Structure Now. How Do We Know If It's Actually Good?

Tonight fixed HOW CodeSentinel decides. It didn't touch whether its decisions are any good. Left on the table, on purpose:

The Reflection critic is itself an LLM judging another LLM — what happens when the critic is wrong, and confidently approves a bad review? We have no metric anywhere in this build for "was this review actually helpful," only "did it get posted." And the evaluate() stub from Chapter 05 needs a real scoring function before Propose-Evaluate-Select means anything — where does that scoring function even come from? Session 5 gives these a name: RAGAS-style evaluation, LLM-as-judge, and the honest limits of both.
