Where we are

You've built BajajBot, DocuMind, HireFlow. These are solid RAG-based systems — LLM reads documents and answers questions. But here's what every one of those systems has in common: you are the agent.

You decide what to retrieve. You decide what action to take. You paste the result. You press send. The LLM just fills in the blanks you set up. That's not an agent — that's autocomplete with memory.

Agentic AI is about flipping that: the system decides, the system acts, the system adapts. You set the goal and walk away.

What just happened — and what didn't

The LLM gave you generic advice. Technically correct. Completely useless in production. Notice what it could not do:

It doesn't know your project. Does your team use SQLAlchemy? Is there a connection pool? Is this code in a payment service? It doesn't know. It's guessing from the snippet you pasted.

It reviewed this file only. Who calls get_user()? Is the result sanitized upstream? Is process_payment() called from a webhook that already validates? Unknown.

It took no action. No GitHub comment. No PR block. No Slack message. No ticket. You have to do all of that yourself — every time, for every file.

Tomorrow, same file, same review. It has no memory. It doesn't know you fixed line 4 last week. It'll tell you again.

What is Agentic AI

Not a chatbot. Not a pipeline. Something that perceives, plans, acts, and remembers — on its own.

The word "agent" gets thrown around a lot. Let's define it precisely so you can think clearly about what you're building — and what you're not.

The Core Idea

In computer science, an agent is anything that can observe its environment, make decisions, and take actions to achieve a goal. That's it. Your robot vacuum is an agent. Your spam filter is an agent. A thermostat is an agent.

What makes an AI agent different is the intelligence layer on top: it can understand natural language, reason about complex goals, plan multi-step actions, use external tools, and adapt when things go wrong.

What makes agentic AI different from a simple AI agent is autonomy over time. You set a goal and step away. It figures out the path.

The Four Pillars — Every Agent Has These

Whether it's your spam filter or Devin, every agent is built from the same four capabilities. The sophistication of each pillar is what separates them.

👁️
Perception

The agent reads and understands its environment. For CodeSentinel: reading PR diffs, file contents, git history, team standards. Not just ingesting text — understanding context.

🧠
Memory

What the agent knows and remembers. Short-term: current investigation. Long-term: "this developer made SQL injection mistakes in 3 past PRs." Without memory, every review starts from zero.

📋
Planning

Deciding what to do next based on what was found. Not a fixed script — a dynamic decision: "This PR touches payment logic, so I check PCI compliance first, then check for race conditions."

🔧
Tools & Actions

The agent acts in the real world. Post a comment on line 47. Block the PR. DM the developer. Create a Jira ticket. Without tools, all that planning is just... text output.

The Key Insight for Developers

An LLM gives you Planning and Perception out of the box (with prompting). What it does not give you by default is Memory that persists across sessions and Tools that act on the real world. Adding those two things is what turns an LLM into an agent.

What Makes It "Agentic" — The Control Loop

Here's the architectural difference that matters most. A simple LLM call is one-shot: prompt → response → done. An agent runs a loop.

👁️Perceive
→
🧠Reason
→
🔧Act
→
📊Observe
→
🔁Repeat

This loop continues until the goal is achieved — or the agent determines it cannot be achieved and escalates. Every step's output feeds the next step's input. The agent adapts based on what it finds.

the_agent_loop.py pseudocode 
# This is the architecture of every agent you'll ever build.
# LangChain handles this loop internally — but understanding
# what it's doing is what makes you dangerous.

from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate

# Step 1: Define what the agent CAN DO (tools)
@tool
def read_file(filepath: str) -> str:
    """Read a source file from the codebase."""
    return open(filepath).read()  # real implementation

@tool
def post_review_comment(filepath: str, line: int, issue: str) -> str:
    """Post a review comment on a specific line in GitHub."""
    return f"Comment posted on {filepath}:{line}"

# Step 2: The agent loop — LangChain manages this for you
llm   = ChatOpenAI(model="gpt-4o")
tools = [read_file, post_review_comment]

agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# Step 3: Give it a GOAL — not instructions
executor.invoke({"input": "Review PR #247. Find issues. Post comments."})

# What LangChain is doing under the hood:
#
# while not done:
#   action = llm.think(goal, scratchpad, tools)
#   if action == TOOL_CALL:
#     result = run_tool(action)
#     scratchpad.append(result)   ← agent sees what it found
#   elif action == FINISH:
#     return result
#
# The PATH is not predetermined — agent decides each step
# based on what the PREVIOUS step returned.

The Evolution — But Grounded in Code Review

Agents didn't appear overnight. Here's how code review itself evolved — and where we're going.

1

Rule-Based: Linters (ESLint, Pylint)

Hard-coded rules. if variable_not_used: raise_warning(). Fast, consistent, deterministic. But completely rigid — no context, no nuance, no understanding of why code exists. Can't reason about business logic. Can't spot architectural issues.

2

Reactive: Static Analysis Tools (SonarQube, Semgrep)

Pattern matching on code structure. Detects known vulnerability patterns. More sophisticated than linters, but still: it sees only what you're looking at. No understanding of surrounding context, no memory of past reviews, no ability to prioritize by actual business risk.

3

GenAI: LLM with a paste (what we just demoed)

Finally understands intent and context. Can reason about code quality holistically. But: no tools, no action, no memory across sessions, no access to your actual project. You are the agent — copy-pasting, acting on advice, repeating for every file.

4

Agentic AI: CodeSentinel (what we're building)

PR created → agent reads all changed files → understands context across the full diff → checks team standards → posts specific comments on exact lines → remembers this developer's past patterns → blocks merge if P0 → notifies via Slack — all without human involvement. That's what agentic means.

GenAI vs AI Agent
vs Agentic AI

Three terms that get conflated constantly. Let's make the distinction precise — with the same code review task at three levels.

Most people use these terms interchangeably. They are not the same. Here's how to think about each one — and why the distinction matters when you're architecting a system.

✏️
GenAI
Answers when asked. Reactive. One prompt, one response. You do all the work around it.
🤖
AI Agent
Uses tools. Takes actions. But you define each task explicitly — still fairly supervised.
🧬
Agentic AI
Pursues goals autonomously. Plans its own path. Handles failures. You set the goal, it delivers.

ai_agent_review.py tools added 
# AI AGENT: LangChain tools + LLM. You trigger per task.
# It acts — but you're still the coordinator.

from langchain_core.tools import tool
from langchain_openai import ChatOpenAI

@tool
def read_file(filepath: str) -> str:
    """Read a source file."""
    return open(filepath).read()

@tool
def post_comment(filepath: str, line: int, comment: str) -> str:
    """Post a review comment on a specific line."""
    return f"Comment posted on {filepath}:{line}"

@tool
def set_pr_status(pr: int, status: str) -> str:
    """Block or approve a PR."""
    return f"PR #{pr} set to {status}"

llm_with_tools = ChatOpenAI(model="gpt-4o").bind_tools(
    [read_file, post_comment, set_pr_status]
)

# You manually trigger each task:
r1 = llm_with_tools.invoke("Review payment_utils.py")
# → LLM decides: call read_file("payment_utils.py")
# → You execute the tool call and return result
# → Done. Waiting for YOUR next instruction.

r2 = llm_with_tools.invoke("Now review auth.py")
# → Another manual trigger. You are still the coordinator.

# bind_tools gave it CAPABILITIES.
# But it has no LOOP, no GOAL, no AUTONOMY.
# You decide what to ask next. Every single time.
What changed

The agent can now act — post comments, set PR status. But it still needs you to decide what to do and when. You've removed the copy-pasting. You haven't removed yourself from the loop.

agentic_review.py autonomous 
# AGENTIC AI: One goal. AgentExecutor runs the loop.
# You get a Slack message when it's done.

from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain.agents import AgentExecutor, create_tool_calling_agent

# All the things CodeSentinel can DO
@tool
def list_pr_files(pr_number: int) -> str:
    """Get all files changed in a pull request."""
    return '["payment_utils.py", "auth.py"]'   # mock

@tool
def read_file(filepath: str) -> str:
    """Read a file from the codebase or standards doc."""
    return open(filepath).read()   # real read

@tool
def post_review_comment(filepath: str, line: int,
                         severity: str, issue: str, fix: str) -> str:
    """Post a review comment on a specific line in GitHub."""
    print(f"📝 [{severity}] {filepath}:{line} — {issue}")
    return "Comment posted"

@tool
def set_pr_status(pr_number: int, status: str, reason: str) -> str:
    """Approve or block the PR from merging."""
    print(f"{'🚫' if status == 'blocked' else '✅'} PR #{pr_number}: {reason}")
    return f"PR {status}"

tools = [list_pr_files, read_file, post_review_comment, set_pr_status]

# The system prompt defines the GOAL and BEHAVIOUR
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are CodeSentinel, an autonomous code reviewer.

When given a PR number:
1. List all changed files
2. Read team_standards.md first
3. Review each file thoroughly  
4. Post specific comments with line numbers and fixes
5. Block PR if any P0 issue found, approve otherwise

Be thorough. Never skip files."""),
    ("human", "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad"),
])

# Wire it up — AgentExecutor IS the loop
llm      = ChatOpenAI(model="gpt-4o")
agent    = create_tool_calling_agent(llm, tools, prompt)
sentinel = AgentExecutor(agent=agent, tools=tools, verbose=True)

# ONE call. Agent handles everything.
sentinel.invoke({"input": "Review PR #247"})

# verbose=True shows the ReAct loop live:
# Thought → Action (tool call) → Observation → Thought → ...
# You'll see the agent making decisions in real time.
What changed

The agent decided which files to check, in what order, what standards to apply, whether to consult past memory, and what action to take — all without your involvement. It adapted its path based on what it found. That's agentic.

Workflow vs Agent
— When to Use Which

The most important architectural decision you'll make. Get this wrong and you'll either over-engineer a simple problem or under-engineer a complex one.

The Honest Truth

Not everything needs an agent. In fact, most things don't. Agents are powerful, but they're also less predictable, harder to test, and more expensive. A workflow that does the same job is always preferable — if the job's steps are known in advance.

The question to ask is not "can I build an agent for this?" — the answer is almost always yes. The question is: "do I know all the steps in advance, or does step 3 depend on what step 2 found?"

Workflows — When the Path Is Fixed

A workflow is a predetermined series of steps. You know step 1, step 2, step 3 before execution starts. The path never changes based on what you find along the way.

workflow_review.py deterministic 
# WORKFLOW: Fixed steps with LCEL — you know this pattern.
# Each chain runs in order. Path never changes.

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(model="gpt-4o")

security_chain = (
    ChatPromptTemplate.from_template("Find security issues:\n{diff}")
    | llm | StrOutputParser()
)
style_chain = (
    ChatPromptTemplate.from_template("Find style issues:\n{diff}")
    | llm | StrOutputParser()
)
report_chain = (
    ChatPromptTemplate.from_template(
        "Summarise review.\nSecurity:{security}\nStyle:{style}"
    )
    | llm | StrOutputParser()
)

def pr_review_workflow(pr_number: int):
    diff     = get_pr_diff(pr_number)                        # Step 1: always
    security = security_chain.invoke({"diff": diff})         # Step 2: always
    style    = style_chain.invoke({"diff": diff})            # Step 3: always
    report   = report_chain.invoke(                          # Step 4: always
        {"security": security, "style": style}
    )
    post_github_comment(pr_number, report)                   # Step 5: always

# Perfectly predictable. Easy to unit test each chain.
# Breaks when PR touches payment module →
# needs PCI check, but no step in this workflow does it.
# You'd add: if "payment" in diff: run_pci_check(diff)
# → Now you're hand-coding agent logic. Time to switch.
Use a workflow when

You can write down every step right now, before seeing any input. The steps don't change based on what you find. Reliability and predictability matter more than flexibility. You need deterministic, auditable output (compliance, finance, healthcare).

Agents — When the Path Is Discovered

An agent figures out what to do next based on what it just found. The path is not predetermined — it emerges from the investigation.

# AGENT: Same entry point — AgentExecutor decides the path.
# verbose=True shows you the decisions in real time.

sentinel = AgentExecutor(
    agent=create_tool_calling_agent(llm, tools, prompt),
    tools=tools,
    verbose=True   # ← shows Thought → Action → Observation
)

# PR #247 — payment code → completely different path:
sentinel.invoke({"input": "Review PR #247"})
# Thought: "payment_utils.py — check PCI compliance first"
# Action:  read_file("team_standards.md")
# Thought: "SQL injection found — check developer history"
# Action:  get_developer_history("rohan@company.com")
# Thought: "3rd time this dev made this mistake → escalate"
# Action:  set_pr_status(247, "blocked", "P0: SQL injection")

# PR #248 — README only → completely different path:
sentinel.invoke({"input": "Review PR #248"})
# Thought: "Only markdown files changed"
# Thought: "No security risk in markdown"
# Action:  set_pr_status(248, "approved", "No code changes")
# Done in 2 steps instead of 8.

# Same AgentExecutor. Same tools. Completely different paths.
# The agent reads the situation and decides.
# No if-else in your code. That's the point.
Use an agent when

"It depends" is the honest answer to "what's step 3?" — the next action depends on what you found in step 2. Tasks require judgment, not just execution. The edge cases are numerous enough that pre-coding all of them is impractical. You're okay with some non-determinism in exchange for much higher capability.

The Decision Framework — Use This in Real Projects

Build a Workflow When

✓ You can write the steps on paper right now
✓ Step N doesn't depend on what step N-1 returned
✓ Same input always produces same execution path
✓ You need 100% reproducible, auditable output
✓ Speed and cost are critical (agents are slower + pricier)

Build an Agent When

✓ The path depends on intermediate findings
✓ The task requires judgment and prioritization
✓ There are too many edge cases to pre-code
✓ You need the system to handle novel situations
✓ The goal is clear but the route is not

The Practical Rule

Start with an LCEL chain (workflow). If you find yourself writing if result_type == "payment": do_extra_check() more than twice — you're hand-coding agent decision logic into a workflow. Switch to AgentExecutor and let the LLM make those conditional decisions. That's exactly what create_tool_calling_agent is built for.

Agentic Patterns — A Quick Preview

Agents aren't monolithic. They use patterns. You'll build all of these across this course — but here's the vocabulary now so concepts land when we get there.

🔗 Chaining — sequential steps where output feeds next input ▼

One LLM call's output becomes the next call's input. Good for: multi-stage refinement (draft → review → polish). Not an agent by itself — it's a workflow with LLM at each step. Becomes agentic when each step decides whether to proceed or branch.

⚡ Parallelization — multiple agents working simultaneously ▼

Security Agent + Performance Agent + Style Agent all review the same PR simultaneously. Faster than sequential. Each agent is specialized. Results are aggregated by a coordinator. You'll build this in Session 5 (MAS).

🔄 Evaluator-Optimizer — generate, evaluate, improve loop ▼

Agent generates a review comment → Evaluator agent checks: "Is this comment specific? Does it have a concrete fix?" → If no: loops back to rewrite. Prevents vague, unhelpful review comments. You'll build this in Session 6 (Design Patterns).

🗺️ Routing — classify first, then send to specialist ▼

Is this PR a security change, a performance change, or a style change? Route to the appropriate specialist agent. The router is a lightweight LLM call. The specialists are optimized for their domain. Avoids one generic agent trying to do everything.

The 12-Session Build Arc

Here's how CodeSentinel grows — one concept per session, one capability added each time.

1

S1 — Foundation (Today) · notebook_01.ipynb

Why LLM alone fails. What agentic AI means. The CodeSentinel vision. You leave with the mental model — no code yet, but you know exactly what you're building and why it needs to be an agent.

2

S2 — Components + PEAS · sentinel_v1.py

PEAS design for CodeSentinel. First working agent: AgentExecutor + 3 tools + real loop. You run it on your own code. It posts comments. It decides whether to block. Every line makes sense because you wrote it.

3

S3 — Memory · sentinel_v2.py + sentinel_v3.py

V1 forgets. Same developer, same bug, same review — no memory. V2 adds episodic memory (past reviews per developer). V3 adds semantic memory (team standards, specialized procedures). Now CodeSentinel remembers patterns.

4

S4 — Architecture + Patterns · sentinel_v4.py

One agent doing everything is slow and shallow. Planner-Executor split: Planner decides strategy, Executor runs checks. Router pattern: payment file → deep security review; markdown file → style only. Different architecture, same tools.

5

S5 — Multi-Agent Systems · sentinel_v5.py

Three specialist agents in parallel: Security Agent + Performance Agent + Style Agent. All reviewing simultaneously. Coordinator synthesizes findings. 3x faster. Results are deeper because each agent specializes.

6

S6 — CrewAI (Learning Tool) · sentinel_crew.py

Rebuild the MAS version using CrewAI to understand Crew, Agent, Task, Process clearly. Note: CrewAI is a learning tool. In production, enterprise teams use frameworks like Microsoft Azure Agent Service (MAF) — SLA-backed, compliance-ready, enterprise-supported. You learn CrewAI so you understand the concepts; you'll recognize them in any framework.

7

S7 — Three C's · Communication, Coordination, Cooperation

How do the Security Agent and Style Agent share findings? What happens when both want to post a comment on the same line? Communication protocols between agents. The concepts behind how agent teams actually coordinate work.

8

S8 — MCP Server · sentinel_mcp_server.py

Right now CodeSentinel runs only from your terminal. MCP makes it callable from anywhere — Claude Desktop, Cursor, VS Code, your team's internal tools. You expose review_pr(), review_file(), get_review_history() as MCP tools. Any AI client can now call CodeSentinel without custom integration. MCP = USB-C for AI tools.

9

S9 — LangGraph · sentinel_v7_langgraph.py

If CodeSentinel crashes mid-review, it starts from scratch. No state persistence. LangGraph fixes this — proper state machine with nodes, edges, conditional routing, and checkpoint persistence. Now CodeSentinel can pause, resume, and recover. Also enables human-in-the-loop: "Should I really block this PR?"

10

S10 — Agentic RAG · sentinel_v8_rag.py

Team standards are static — but your codebase changes every day. Agentic RAG: agent decides what to retrieve based on what it's finding. "This file uses async — let me retrieve the team's async patterns doc." Iterative retrieval, self-RAG, query rewriting. Your RAG knowledge from Module 2 upgrades here.

11

S11 — Full System · sentinel_v9_full.py

LangChain (retrieval layer) + LangGraph (orchestration layer) combined. Single-agent and multi-agent supervisor pattern. Shared state management between agents. This is the production-grade version — the one that goes on your GitHub and in your interviews.

12

S12 — Eval + Debug · sentinel_eval.py

Did the agent find the real root cause? Did it falsely block a clean PR? LLM-as-judge for review quality. LangSmith trace analysis. Where did reasoning break? How do you catch infinite loops in production? This is what separates engineers who ship agents from those who demo them.
