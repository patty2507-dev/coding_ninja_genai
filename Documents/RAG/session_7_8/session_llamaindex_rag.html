Why LlamaIndex
When We Have LangChain?

Context

We built BajajBot across 5 sessions — PDF ingestion, ChromaDB, Pinecone, FastAPI, and an LCEL chain. That pipeline works. But you've probably noticed that every time you want to change the retrieval strategy, or add metadata filtering, or swap your vector store, you're rewriting significant logic.

LlamaIndex was built with exactly that problem in mind. Its primary design goal is to be the data framework for LLM applications — it goes much deeper into the ingestion and indexing layer than LangChain does.

The Honest Framing

LangChain and LlamaIndex are not competitors. They solve overlapping but distinct problems. Understanding where each shines is what separates a senior AI engineer from someone who just copies boilerplate.

🔗 LangChain's Strength

Orchestration and chains. Building complex workflows, tool use, agents, and connecting diverse APIs together. The "plumbing" between components.

🦙 LlamaIndex's Strength

Data ingestion, indexing, and retrieval. It knows your documents at a much deeper level — metadata, relationships between nodes, hierarchical retrieval, graph structures.

The Real-World Problem

You built BajajBot. Users ask: "What is the penalty for late EMI payment under Scheme B, specifically for loans under ₹10 lakh?"

Your LangChain+ChromaDB pipeline returns 3 chunks about EMI — none of which answer the full question because the penalty table and scheme definition are in different sections of the document, separated by 40 pages.

LangChain doesn't know your document structure. LlamaIndex does.

LlamaIndex's Mental Model

Think of it this way: LangChain gives you chains and tools. LlamaIndex gives you a data engine. It understands documents as structured objects, not just raw text blobs.

📄
Documents
Raw files with rich metadata. PDFs, web pages, SQL tables, APIs — all normalized.
🔷
Nodes
Chunks with relationships. Each node knows its parent, siblings, and source section.
🗂️
Indexes
Multiple index types — vector, summary, knowledge graph. Not just one strategy.
🔍
Retrievers
Pluggable retrieval — semantic, BM25, hybrid, auto-retrieval with metadata filters.
⚙️
Query Engine
End-to-end Q&A over your data. One API call, full RAG pipeline.
🔄
Workflows
Event-driven orchestration for multi-step agentic pipelines. LlamaIndex's answer to LangGraph.

Core Concepts

The primitives you'll use every day — directly from the official LlamaIndex docs, with production context added.

The Primitive Stack

Before writing a single line, you need this mental model locked in. Every LlamaIndex application is built from these 5 primitives stacked on each other.

1

LLM — The Brain

The foundation. LlamaIndex wraps any LLM provider (OpenAI, Azure OpenAI, Anthropic, local Ollama models) through a unified LLM interface. You configure it once globally via Settings.llm and every component downstream uses it automatically.

llm_setup.py LlamaIndex way 
# LlamaIndex global settings — configure once, applies everywhere
from llama_index.core import Settings
from llama_index.llms.azure_openai import AzureOpenAI
from llama_index.embeddings.azure_openai import AzureOpenAIEmbedding

Settings.llm = AzureOpenAI(
    model="gpt-4o",
    deployment_name="gpt-4o-deployment",
    api_key=os.environ["AZURE_OPENAI_KEY"],
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_version="2024-02-01",
)

Settings.embed_model = AzureOpenAIEmbedding(
    model="text-embedding-3-large",
    deployment_name="text-embedding-3-large",
    api_key=os.environ["AZURE_OPENAI_KEY"],
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
)

Settings.chunk_size = 512
Settings.chunk_overlap = 50

# From this point, ALL indexes, retrievers, query engines inherit these settings.
# No need to pass llm=... to every single component like in LangChain.
2

Document — The Raw Input

A Document is a container for any data source. It's not just text — it carries metadata that persists through chunking. In LangChain, Document metadata is often lost or inconsistently propagated. In LlamaIndex, metadata is a first-class citizen.

documents.py 
from llama_index.core import SimpleDirectoryReader, Document

# SimpleDirectoryReader — LlamaIndex's equivalent of LangChain loaders
# Handles PDF, DOCX, CSV, images, audio — with auto file type detection
documents = SimpleDirectoryReader(
    input_dir="./data/bajaj_documents",
    recursive=True,
    filename_as_id=True,  # Document ID = filename — critical for updates
    required_exts=[".pdf", ".txt"],
).load_data()

# Or create a Document manually with full metadata control
doc = Document(
    text="Personal loan interest rate is 14% p.a. for salaried employees.",
    metadata={
        "product": "personal_loan",
        "section": "interest_rates",
        "effective_date": "2024-01-01",
        "source": "bajaj_product_guide_v3.pdf",
    },
    # This metadata filters down to EVERY chunk created from this document
    metadata_separator=";",
    excluded_llm_metadata_keys=["source"],  # Don't pollute LLM context with this
    excluded_embed_metadata_keys=["effective_date"],  # Don't embed this field
)
3

Node — The Chunk with a Brain

When a Document gets chunked, each chunk becomes a TextNode. Unlike LangChain's simple Document chunks, Nodes carry relationships: which other nodes came before and after, what their source document is, and optionally their position in a hierarchy (for recursive retrieval).

nodes.py 
from llama_index.core.node_parser import SentenceSplitter, HierarchicalNodeParser
from llama_index.core.schema import TextNode, NodeRelationship, RelatedNodeInfo

# Standard sentence-aware chunking (smarter than naive character split)
parser = SentenceSplitter(chunk_size=512, chunk_overlap=50)
nodes = parser.get_nodes_from_documents(documents)

# Each node carries: id, text, metadata, relationships
print(nodes[0].node_id)           # UUID auto-assigned
print(nodes[0].metadata)          # Inherited from parent Document
print(nodes[0].relationships)     # {NEXT: node_id_of_next_chunk, SOURCE: doc_id}

# ── HIERARCHICAL NODES — the real power ──
# Creates parent-child node trees. Parent = whole section. Child = sentence.
# Retrieval finds the child (precise), but returns the parent (contextual).
hier_parser = HierarchicalNodeParser.from_defaults(
    chunk_sizes=[2048, 512, 128]  # Coarse → medium → fine
)
hier_nodes = hier_parser.get_nodes_from_documents(documents)
# Result: nodes[i].parent_node links fine chunks to their section context
💡 Why This Matters

In LangChain, if you chunk a document, you lose the "this paragraph belongs to Section 3.2" information. In LlamaIndex, every node carries its lineage. This enables recursive retrieval: find the specific sentence, but return the whole section as context. That's how you solve the "penalty table is 40 pages away from the scheme definition" problem.

4

Index — Multiple Strategies, One Interface

This is the core differentiator. LlamaIndex has multiple Index types, each with a different retrieval strategy under the hood. The query API is identical across all of them.

📊
VectorStoreIndex

Semantic similarity search. Your BajajBot uses this. Backed by Pinecone, ChromaDB, Azure AI Search, etc.

📝
SummaryIndex

Summarizes all nodes sequentially. Best for "summarize this 200-page report" use cases.

🌐
PropertyGraphIndex

Knowledge graph extraction. Entities and relationships. Best for complex multi-hop reasoning.

🏗️
ComposableGraph

Index over multiple indexes. Query across PDFs, databases, and APIs simultaneously.

5

QueryEngine — The High-Level Interface

A QueryEngine wraps Retriever + PostProcessor + ResponseSynthesizer into one call. It's what you actually expose to users. LangChain's equivalent is a retrieval chain, but QueryEngine has more built-in response modes.

Phase 1: Ingestion Pipeline

LlamaIndex has a dedicated IngestionPipeline class — equivalent to what you'd manually chain together in LangChain. The key advantage: caching and incremental updates are built in.

ingestion_pipeline.py Production Pattern 
from llama_index.core import SimpleDirectoryReader
from llama_index.core.ingestion import IngestionPipeline, IngestionCache
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.extractors import TitleExtractor, QuestionsAnsweredExtractor
from llama_index.vector_stores.pinecone import PineconeVectorStore

# Load raw documents
documents = SimpleDirectoryReader("./bajaj_docs").load_data()

# Build the ingestion pipeline — transformations applied sequentially
pipeline = IngestionPipeline(
    transformations=[
        # Step 1: Chunk documents into nodes
        SentenceSplitter(chunk_size=512, chunk_overlap=50),

        # Step 2: Auto-extract metadata using LLM
        # This enriches each node with: title of section, questions this chunk answers
        TitleExtractor(nodes=5),       # Infers section title per chunk
        QuestionsAnsweredExtractor(questions=3),  # What 3 Q's does this answer?

        # Step 3: Embed — uses Settings.embed_model automatically
        # (OpenAIEmbedding, AzureOpenAIEmbedding, HuggingFaceEmbedding, etc.)
        Settings.embed_model,
    ],
    vector_store=PineconeVectorStore(pinecone_index=pinecone_index),
    cache=IngestionCache(),  # Caches transform results — re-run is a no-op for unchanged docs
)

# Run ingestion — returns nodes
nodes = await pipeline.arun(documents=documents)  # async-first for production
print(f"Ingested {len(nodes)} nodes")
🔥 LangChain Comparison

In LangChain, you'd manually call text_splitter.split_documents(), then loop to call embeddings.embed_documents(), then call vectorstore.add_documents(). The IngestionPipeline wraps all of this with caching, async, and metadata extraction built in. No more "did this document already get ingested?" bugs in production.

Phase 2: The Query Pipeline

When a user asks a question, LlamaIndex breaks the process into three explicit stages. You can intercept and customize each one.

1

Retriever — Finding the Right Nodes

By default, LlamaIndex uses vector similarity to find the top-k nodes. But you can swap this for BM25, hybrid search, or auto-retrieval (where the LLM generates metadata filters dynamically).

retriever.py 
from llama_index.core import VectorStoreIndex
from llama_index.core.vector_stores import MetadataFilter, MetadataFilters, FilterOperator

index = VectorStoreIndex.from_vector_store(vector_store)

# Standard retriever — semantic similarity, top-5
retriever = index.as_retriever(similarity_top_k=5)

# Production retriever — semantic + metadata filter
# Only retrieve nodes from "interest_rates" section, for personal loans
retriever_filtered = index.as_retriever(
    similarity_top_k=5,
    filters=MetadataFilters(filters=[
        MetadataFilter(key="product", value="personal_loan"),
        MetadataFilter(key="section", value="interest_rates", operator=FilterOperator.EQ),
    ])
)

nodes = retriever_filtered.retrieve("What is the interest rate for ₹5L loan?")
for node in nodes:
    print(f"Score: {node.score:.3f} | Source: {node.metadata['source']}")
    print(node.text[:200])
2

Node Postprocessors — Reranking and Filtering

After retrieval, postprocessors clean and re-rank the results before they hit the LLM. This is where you insert Cohere reranker, LLM-based reranking, or duplicate filtering.

🚫 No External API Needed

We'll use LlamaIndex's built-in postprocessors — LLMRerank, SentenceTransformerRerank, and KeywordNodePostprocessor. Zero extra API keys. Three different trade-offs shown below.

LLMRerank — Uses your existing LLM to score each retrieved node's relevance. No extra library. Best quality. Costs a few extra tokens per query. Recommended starting point.

rerank_llm.pyZero extra deps
from llama_index.core.postprocessor import SimilarityPostprocessor, LLMRerank

# Stage 1: drop anything below 0.65 cosine similarity
# Stage 2: ask the LLM to pick the best 3 from remaining
postprocessors = [
    SimilarityPostprocessor(similarity_cutoff=0.65),
    LLMRerank(
        choice_batch_size=5,   # Score 5 at a time
        top_n=3,              # Return top 3 after reranking
        choice_select_prompt=None,  # Uses built-in BFL-friendly prompt
    ),
]

# LLMRerank prompt effectively asks:
# "Given this query about BFL products, rank these passages by relevance."
# Uses Settings.llm — no extra API key, reuses your Azure OpenAI deployment.

query_engine = RetrieverQueryEngine(
    retriever=index.as_retriever(similarity_top_k=8),
    node_postprocessors=postprocessors,
    response_synthesizer=get_response_synthesizer(response_mode="compact"),
)

SentenceTransformerRerank — Downloads a small cross-encoder model locally (e.g. cross-encoder/ms-marco-MiniLM-L-2-v2, ~80MB). Fastest at inference, no API cost per query. Good for high-volume production.

rerank_local.pypip install sentence-transformers
from llama_index.core.postprocessor import SentenceTransformerRerank

# Cross-encoder: takes (query, passage) pair and scores relevance directly
# Much more precise than bi-encoder similarity used in vector retrieval
reranker = SentenceTransformerRerank(
    model="cross-encoder/ms-marco-MiniLM-L-2-v2",  # ~80MB, fast
    top_n=3,
)

# For BFL use-case, retrieve 10 candidates, rerank to top 3 with cross-encoder
postprocessors = [
    SimilarityPostprocessor(similarity_cutoff=0.60),
    reranker,
]

# Why this works better than pure vector retrieval:
# Vector: "personal loan rate" matches "consumer credit interest" (semantic)
# Cross-encoder: scores the actual (query, passage) pair — catches context mismatches

KeywordNodePostprocessor — Zero ML, zero cost. Filters retrieved nodes to only those containing required keywords. Good as a hard pre-filter before semantic reranking — e.g. always require "EMI" to appear in nodes when the user asks about EMI.

rerank_keyword.pyZero cost
from llama_index.core.postprocessor import KeywordNodePostprocessor

# Keep only nodes that contain at least one required keyword
# Exclude nodes with any forbidden keyword
postprocessors = [
    KeywordNodePostprocessor(
        required_keywords=["EMI", "instalment"],  # At least one must appear
        exclude_keywords=["deprecated", "discontinued"],  # Must NOT appear
    ),
    LLMRerank(top_n=3),  # Then LLM picks the best 3 from keyword-filtered set
]

# Pattern: Keyword filter (cheap, fast) → LLMRerank (precise) → response synthesis
# This is a 3-stage funnel: retrieve 8 → keyword filter → LLM pick 3 → answer
3

Response Synthesizer — LlamaIndex's Hidden Superpower

This is what LangChain doesn't have out of the box. LlamaIndex has multiple response modes you can swap with one line.

refine (default): Processes each retrieved node one at a time, refining the answer iteratively. LLM call per node. Best quality for complex questions. Most tokens used.

response_mode="refine"

compact: Packs as many nodes as possible into a single prompt, then refines if needed. Best balance of quality and cost. Recommended for production.

response_mode="compact"

tree_summarize: Builds a tree of answers — generates answers per chunk, then merges them bottom-up. Best for summarization over very long documents (100+ pages).

response_mode="tree_summarize"

accumulate: Generates independent answers for each retrieved node, then concatenates. Good for multi-document comparison. Returns a structured list of answers.

response_mode="accumulate"

Index Types — How Each One Thinks

LlamaIndex has 4 production-relevant index types. They all accept the same Document input and expose the same query API — but what they build internally is completely different. Choose wrong and your retrieval quality collapses.

retrieval: ANN similarity best for: specific Q&A scales to: millions of nodes

BFL Use Case

"What documents are required for a ₹10L personal loan?" — VectorStoreIndex finds the exact nodes mentioning documentation requirements. This is your BajajBot's primary index. Works best when questions are specific and the answer lives in one or two chunks.

retrieval: sequential, all nodes best for: summarisation response_mode: tree_summarize

BFL Use Case

"Summarize BFL's key financial highlights from the 2024 annual report." — SummaryIndex reads every node in sequence and uses tree_summarize to build a coherent answer. Not for point-in-time queries — for full-document comprehension tasks. Think investor briefings, audit summaries.

retrieval: inverted keyword index best for: exact term lookup no embedding cost

When to use

When your users search for exact product codes, legal clause numbers, or domain-specific terms that embedding models might not understand well. BFL example: searching by "Section 12B", "NACH mandate", or "Scheme Code PL-201" — pure keyword lookup beats semantic search here. Also useful as a hybrid layer alongside VectorStoreIndex.

retrieval: AutoMerging best for: context-rich Q&A solves: "lost in the middle"

BFL Use Case — The Hard Problem

"What is the penalty for EMI bounce, and does it affect my CIBIL?" — The answer requires the EMI fee (a Level-2 fine-grained chunk) AND the credit reporting policy (nearby context, only in the Level-1 section). AutoMerging retrieves the precise child node, but returns the parent for full context. Solves the "penalty table is 40 pages from scheme definition" problem natively.

Multi-Query Retriever — Smarter Retrieval, Zero Extra APIs

A single embedding query often misses relevant nodes because users phrase things in one way, but documents use different vocabulary. MultiQueryRetriever solves this by generating multiple query variants using your LLM, running parallel retrievals, and deduplicating the results — all within LlamaIndex, no Cohere or external reranker needed.

multi_query_retriever.py Zero extra API keys 
from llama_index.core.retrievers import QueryFusionRetriever
from llama_index.core.postprocessor import LLMRerank
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.response_synthesizers import get_response_synthesizer

# ── Step 1: Multi-query retriever (LlamaIndex's QueryFusionRetriever) ──
# Generates N query variants, retrieves from each, deduplicates results
retriever = QueryFusionRetriever(
    retrievers=[
        index.as_retriever(similarity_top_k=5),  # One retriever, multiple queries
    ],
    similarity_top_k=5,
    num_queries=3,             # Generate 3 query variants using Settings.llm
    mode="reciprocal_rerank",   # Fusion algo: RRF (no ML, just ranking math)
    use_async=True,             # Parallel retrieval for the 3 queries
    verbose=True,               # Prints generated queries — great for debugging
)

# ── Step 2: LLMRerank as final cleanup ──
# After fusion, LLM picks the best 4 from fused candidates
postprocessors = [LLMRerank(choice_batch_size=5, top_n=4)]

# ── Step 3: Wrap into query engine ──
query_engine = RetrieverQueryEngine(
    retriever=retriever,
    node_postprocessors=postprocessors,
    response_synthesizer=get_response_synthesizer(response_mode="compact"),
)

# ── Test with a BFL query ──
response = query_engine.query(
    "What are all the charges and fees for a BFL personal loan?"
)
# Generated queries (verbose output):
#   Q1: "Bajaj Finance personal loan processing fee"
#   Q2: "BFL loan charges interest rate"
#   Q3: "personal loan origination cost documentation fee"
# All 3 retrieve different nodes → fusion gives broader, more complete coverage

print(response)
for n in response.source_nodes:
    print(f"[{n.score:.2f}] {n.metadata.get('section','?')} | {n.text[:120]}")
💡 Reciprocal Rank Fusion (RRF)

RRF combines results from multiple retrievals using a pure math formula: score = Σ 1/(k + rank_i) where k=60 by default. A node that appears in the top-5 of all 3 query retrievals gets a very high fused score — without any ML, without any API key, just rank arithmetic. It's the same algorithm used in enterprise search systems at scale.

Document Management — The Critical Production Feature

In production, documents change. LlamaIndex has a built-in document management system that tracks what's already indexed and only re-processes changed files. LangChain has no equivalent.

Production Reality Check

At AstraZeneca/Rolls-Royce scale, documents update constantly — clinical trial protocols, gate review templates, product specs. Without document tracking, you either re-index everything on every update (expensive) or have stale vectors silently returning wrong answers.     SimilarityPostprocessor(similarity_cutoff=0.65),

    structured_answer_filtering=True,  # Filter out irrelevant answers

LangChain vs LlamaIndex
The Real Decision Guide

Not a "which is better" debate. A systematic framework for choosing the right tool for each architectural layer.

How to Think About This

They're not mutually exclusive. Anthropic's own teams use both. LlamaIndex for data ingestion and indexing, LangChain/LangGraph for orchestration. You can use LlamaIndex's QueryEngine as a tool inside a LangChain agent. Pick by layer, not by religion.

Feature-by-Feature Comparison

Concern LangChain LlamaIndex Winner
Document ingestion Loaders + text splitters, manual IngestionPipeline with caching, metadata extraction LlamaIndex
Vector store integration Good, but basic Deeper: metadata filtering, document tracking LlamaIndex
Index strategies Mostly vector similarity Vector, Summary, Knowledge Graph, Hierarchical LlamaIndex
Agent framework LangGraph — mature, production-tested Workflows — newer, event-driven LangChain
Tool/function calling Comprehensive tool ecosystem Growing, less mature LangChain
Response synthesis modes Manual (stuff, map-reduce, refine) Built-in: refine, compact, tree_summarize, accumulate LlamaIndex
Document updates Manual: delete by ID, re-insert update_ref_doc, insert, delete_ref_doc LlamaIndex
Streaming Yes Yes Tie
Human-in-the-loop LangGraph: interrupt, human approval Workflow: InputRequiredEvent LangChain
Multi-agent patterns LangGraph: mature, supervisor pattern LlamaAgents — newer LangChain
Observability LangSmith (paid) Arize Phoenix, TruLens (open source) LlamaIndex
Learning curve Steeper — more abstractions More opinionated — easier for RAG LlamaIndex

When to Use Which — The Decision Tree

🏗️ Architectural Decision Framework
IF building RAG
→
Start with LlamaIndex. Better ingestion, multiple index types, built-in response synthesis modes. Your BajajBot, Rolls-Royce Gate Review, AstraZeneca experiment tracking — all would benefit from LlamaIndex's data layer.
IF building agents
→
Start with LangGraph. More mature, battle-tested in production, better human-in-the-loop patterns, larger community.
IF documents update frequently
→
LlamaIndex without question. update_ref_doc, insert, IngestionCache — document management is native, not bolted on.
IF you need multi-hop reasoning over a corpus
→
LlamaIndex's PropertyGraphIndex or Hierarchical Nodes. LangChain doesn't have native equivalents.
IF orchestrating 3+ tools + memory + planning
→
LangGraph. Supervisor agents, checkpointing, interrupt/resume — LangGraph is purpose-built for this. LlamaIndex Workflows is catching up.
IF it's a hybrid system
→
Both. LlamaIndex QueryEngine as a LangChain tool, or LangGraph as the orchestrator calling LlamaIndex for retrieval. This is the enterprise architecture pattern.

The Hybrid Architecture Pattern

This is how production systems at scale actually work. LlamaIndex owns the data layer, LangGraph owns the orchestration layer.

LlamaIndex Workflows

LlamaIndex's answer to LangGraph — event-driven, async-first orchestration for complex RAG pipelines.

🔍 Context

You know LangGraph — stateful graphs with nodes and edges. LlamaIndex Workflows take a different approach: event-driven. Steps communicate by emitting and consuming typed events. No explicit edge definitions. More flexible for complex, branching pipelines.

Workflow Concepts

A Workflow is a class with decorated step functions. Each step has an input event type and emits an output event type. The framework wires everything together at runtime.

🔗 LangGraph Mental Model

Define a graph with nodes. Connect nodes with explicit edges. State flows through the graph. Control flow = graph structure.

🦙 Workflow Mental Model

Define steps with event signatures. Steps emit events, other steps consume them. Control flow = event routing. More composable for complex branching.

rag_workflow.py Advanced RAG Pipeline 
from llama_index.core.workflow import (
    Workflow, step, Event, StartEvent, StopEvent, Context
)
from llama_index.core import VectorStoreIndex
from pydantic import BaseModel

# ── Define event types (the "messages" between steps) ──
class QueryEvent(Event):
    query: str
    product_type: str

class RetrievalEvent(Event):
    nodes: list
    query: str

class RerankedEvent(Event):
    nodes: list
    query: str

# ── Build the Workflow ──
class BajajRAGWorkflow(Workflow):

    def __init__(self, index: VectorStoreIndex, **kwargs):
        super().__init__(**kwargs)
        self.index = index

    @step
    async def route_query(self, ctx: Context, ev: StartEvent) -> QueryEvent:
        """Step 1: Classify the query and extract product type."""
        user_query = ev.query
        # Use LLM to extract product type from query
        product = await ctx.llm.acomplete(
            f"From this query, extract the product type (personal_loan, home_loan, emi_card, other):\n{user_query}"
        )
        return QueryEvent(query=user_query, product_type=str(product).strip())

    @step
    async def retrieve(self, ctx: Context, ev: QueryEvent) -> RetrievalEvent:
        """Step 2: Retrieve with metadata filter based on product type."""
        from llama_index.core.vector_stores import MetadataFilter, MetadataFilters
        filters = None
        if ev.product_type != "other":
            filters = MetadataFilters(filters=[
                MetadataFilter(key="product", value=ev.product_type)
            ])
        retriever = self.index.as_retriever(similarity_top_k=8, filters=filters)
        nodes = await retriever.aretrieve(ev.query)
        return RetrievalEvent(nodes=nodes, query=ev.query)

    @step
    async def rerank(self, ctx: Context, ev: RetrievalEvent) -> RerankedEvent:
        """Step 3: Rerank with LLMRerank — no extra API keys."""
        from llama_index.core.postprocessor import LLMRerank
        reranker = LLMRerank(choice_batch_size=5, top_n=4)
        reranked = reranker.postprocess_nodes(ev.nodes, query_str=ev.query)
        return RerankedEvent(nodes=reranked, query=ev.query)

    @step
    async def synthesize(self, ctx: Context, ev: RerankedEvent) -> StopEvent:
        """Step 4: Generate response."""
        from llama_index.core.response_synthesizers import get_response_synthesizer
        synth = get_response_synthesizer(response_mode="compact")
        response = await synth.asynthesize(ev.query, nodes=ev.nodes)
        return StopEvent(result=response)

# ── Run the Workflow ──
workflow = BajajRAGWorkflow(index=bajaj_index, timeout=60, verbose=True)
result = await workflow.run(query="What is the EMI for a ₹5L personal loan over 36 months?")
print(result)

If you're already invested in LangGraph (BajajBot Session 6), don't rewrite everything. Use LangGraph for orchestration and LlamaIndex for the retrieval steps inside your graph nodes. If you're starting a new LlamaIndex-native project, Workflows give you async event routing without having to define a graph schema upfront.

streaming_workflow.py
# Workflows support streaming — emit a stream event from the synthesize step
from llama_index.core.workflow import StreamEvent

@step
async def synthesize(self, ctx: Context, ev: RerankedEvent) -> StopEvent:
    from llama_index.core.response_synthesizers import get_response_synthesizer
    synth = get_response_synthesizer(response_mode="compact", streaming=True)
    stream = await synth.asynthesize(ev.query, nodes=ev.nodes)
    async for token in stream.async_response_gen():
        ctx.write_event_to_stream(StreamEvent(delta=token))
    return StopEvent(result=stream.response_txt)

# In FastAPI endpoint:
async def stream_query(query: str):
    handler = workflow.run(query=query)
    async for event in handler.stream_events():
        if isinstance(event, StreamEvent):
            yield event.delta
